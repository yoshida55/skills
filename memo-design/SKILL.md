# memo-design スキル

今回の会話・作業の中で行ったデザイン判断を `50_design.md` に追記するスキル。
コードではなく「なぜそうしたか」という理由・経緯を残す。

## 使い方
```
/memo-design
```

---

## 実行フロー

### Step 1: 保存先ファイルを確認する
- 自宅PC: `D:\50_knowledge\50_design.md`
- 会社PC: `C:\Users\guest04\Desktop\高橋研三\03_knowledge\50_design.md`

判定:
```bash
test -f "D:/50_knowledge/50_design.md" && echo "home" || echo "company"
```

### Step 2: 今回の作業からデザイン判断を抽出する

会話の中から以下の観点で判断を洗い出す：

- **選んだ・選ばなかった理由**（画像・色・レイアウト・アニメーション）
- **数値の根拠**（opacity・duration・delay・remの値など）
- **ターゲットを意識した判断**（誰に見せるか・何を伝えるか）
- **やりすぎを止めた判断**（シンプルにした・削除した理由）
- **比較して選んだ判断**（AよりBにした理由）

### Step 3: ユーザーに確認する（省略可）

抽出した判断を箇条書きで出して「これで合ってますか？」と確認してもよい。
ユーザーが「つづけて」と言ったらそのまま書き込む。

### Step 4: ファイルに追記する

- `50_design.md` を Read で読み込む
- 末尾に追記する（既存内容は絶対に消さない）
- **Edit ツールで直接追記**

---

## 保存フォーマット

```markdown
## YYYY-MM-DD プロジェクト名

### 判断のタイトル
- **決定**: 何をしたか
- **理由**: なぜそうしたか
- **教訓**: 次に活かせること（任意）
- **数値メモ**: 具体的な値（任意）
```

---

## ルール

- **コードは書かない** → コードはソースを見ればわかる。理由だけを残す
- **「正解かどうか」は問わない** → その時点での判断理由を残す。後から更新してよい
- **プロジェクト固有すぎる内容は一般化する** → 後から別案件でも使えるように書く
- **既存の内容は絶対に上書きしない**
- 同じ日・同じプロジェクトの追記は同じ `##` セクションに加える

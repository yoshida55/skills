# review-fix スキル

`review-css` スキルが出力した `⚠ REVIEW:` コメントを読んで、
- **自動修正できるもの** → 確認なしで即修正
- **判断が必要なもの** → ユーザーに1件ずつ確認してから修正

コメント形式はファイルによって異なる：
- CSS: `// ⚠ REVIEW: ...`
- HTML: `<!-- ⚠ REVIEW: ... -->`

---

## 🎯 対象ファイルの決定ルール

1. **会話の中で最後に登場した `ide_opened_file` のファイル**を対象とする
   - `ide_opened_file` が複数ある場合は**一番下（最新）のもの**を使う
   - 間違えないように、作業開始前に「対象ファイル: ○○.php」と必ず明示する
2. 引数で明示されている場合はそちらを優先する（例: `/review-fix css/work.css`）
3. 不明な場合はユーザーに確認する

---

## 🔗 横断的チェックのルール

対象ファイルがCSSの場合、**HTML・PHPファイルも同時にチェック・修正する**。
対象ファイルがHTMLの場合、**CSSも同時にチェック・修正する**。

| 問題の種類 | 修正対象ファイル |
|---|---|
| クラス名のスペルミス | CSS + 同じクラスを使っているHTML/PHPファイル全て |
| クラス名の変更 | CSS + HTML/PHP全て |
| HTMLタグの誤り | HTML/PHP のみ |
| CSSプロパティの問題 | CSS のみ |

横断チェックの手順：
1. 対象ファイルを読む → `⚠ REVIEW:` を全件抽出（CSS: `//`, HTML: `<!-- -->`）
2. クラス名に関わる修正がある場合 → Grep でプロジェクト内のHTML/PHP/CSSファイルを検索
3. 該当ファイルを読み込んで同時修正

> ⚠ **注意**: ユーザーがREVIEWコメント行だけ手動で削除している場合がある。
> コメントが消えていても問題自体は残っているケースがあるため、
> ファイルを読んだ際に明らかに怪しい箇所（不要なラッパー・hrefなし等）は
> REVIEWコメントがなくても報告・確認すること。

---

## 🎯 実行フロー

### Step 0（必須）: TodoWrite でチェックリストを出す
以下の順で必ず実行する：
1. IDEで開いているファイル（または指定ファイル）を読み込む
2. `⚠ REVIEW:` コメントを全件リストアップ（CSSは `// ⚠ REVIEW:`、HTMLは `<!-- ⚠ REVIEW: -->`）
3. 自動修正リスト・確認リストに分類
4. クラス名に関わる修正がある場合はCSS/HTML/PHPも横断検索する
5. 自動修正を実行（対象ファイル + 関連ファイルに反映）
6. 確認リストをユーザーに提示して回答をもらう
7. ユーザーの回答に従って修正

---

## 🔍 REVIEWコメントの分類ルール

### ✅ 自動修正してよいもの（確認不要）
| パターン | 対応 |
|---|---|
| スペルミス（英語クラス名の誤字） | 正しいスペルに修正（CSS・HTML両方同時に修正） |
| AIっぽいコメント・変更履歴コメントの削除 | 該当行を削除 |
| `display: block` をブロック要素に指定 → 不要 | 削除 |
| `width: 100%` をブロック要素に指定 → 不要 | 削除 |
| 同一セレクタの重複 → マージ | マージして1つにまとめる |
| `// ⚠ REVIEW:` / `<!-- ⚠ REVIEW: -->` コメント行自体 | 修正後に必ず削除する |
| デフォルト値の指定（`data-aos-delay="0"` 等） | 該当属性を削除 |
| 同一属性の重複定義（例: `href="#"` が2つ） | 重複を1つにまとめる |

### ❓ ユーザーに確認が必要なもの
| パターン | 確認内容 |
|---|---|
| 数値の変更（px→rem変換、max-widthなど） | 変換後の値を提示してYes/Noを聞く |
| CSS変数化の提案 | 変数化するか聞く |
| 共通化・クラス整理 | 具体的な修正案を見せてYes/Noを聞く |
| 削除すると影響範囲が大きいもの | 影響を説明してYes/Noを聞く |
| `position: relative` 等、状況次第のもの | 用途を確認してから判断 |
| 不要なラッパー `<div>` の削除 | 削除後の構造を見せてYes/Noを聞く |
| `<a>` の `href` がない → リンク追加 | どのURLにするか確認してから修正 |
| 仮データ・仮クラス名の置き換え | 正しい値を確認してから修正 |

## 📋 出力フォーマット

### 自動修正完了の報告
```
✅ 自動修正済み（N件）
- スペルミス: `.discripsion` → `.description`（CSS + HTML）
- display: block 削除: `.header_logo`
- REVIEWコメント行を削除
```

### ユーザーへの確認
```
❓ 確認が必要な項目（M件）

【1/M】 max-width の単位
現状: max-width: 1360rem;
提案: max-width: 136rem; （10px = 1rem 換算で 1360px → 136rem）
💬 Claudeの意見: remで管理しているなら136remに揃えた方が一貫性があります。修正を推奨します。
→ 修正しますか？ [A: する / B: しない]

【2/M】 ...
```

### 💬 Claudeの意見の書き方ルール
- 1〜2行で簡潔に
- 「推奨 / どちらでもよい / プロジェクトによる」のどれかを必ず含める
- 理由も1行で添える
- 例：
  - `💬 Claudeの意見: セマンティック的に正しいので推奨します。CSSへの影響もありません。`
  - `💬 Claudeの意見: どちらでもよいですが、他ページと揃えるなら変更する価値あり。`
  - `💬 Claudeの意見: プロジェクトの規模次第。小規模なら変数化しなくてもOKです。`

---

## 注意事項
- `// ⚠ REVIEW:`（CSS）と `<!-- ⚠ REVIEW: -->`（HTML）、どちらも対象にする
- REVIEWコメント行は修正後に必ず削除する（残さない）
- スペルミス修正はCSS・HTML両方同時に修正する
- 自動修正は確認なしで直接ファイルに反映する
- 確認項目はまとめて一覧で提示する（1件ずつ返信を待たない）
- ユーザーが「A」「B」「C」等で回答したら対応する
- 修正件数ゼロでも「REVIEWコメントなし」と報告する
